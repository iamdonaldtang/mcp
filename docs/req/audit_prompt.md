# 需求文档完整性审计提示词

> 用途: 将此提示词交给 AI 或人工审计员，对照 `req_community.md` 和 `req_white_label.md` 逐页检查以下 6 个维度的完整性。
> 日期: 2026-03-07

---

## 审计指令

请逐页审计以下两份前端需求文档，检查每个页面是否在 **6 个维度** 上都有明确、无歧义的规格说明。如有缺失或含糊，按页面列出缺失项，标注优先级（P0 阻塞开发 / P1 影响体验 / P2 完善度）。

**审计范围**:
- `docs/req/req_community.md` — Community 产品全部页面（B09-B12 Hub, B13/B34/B35/B55 Wizard, B31/B31a-B31i 模块管理, B49/B50 设置, B32/B33/B54/B61 运营, D01-D11/D16-D20 Modal）
- `docs/req/req_white_label.md` — White Label 产品全部页面（B14-B16 Hub, B37/B17/B57-B60/B38/B56 Wizard, B18-B26/B40-B43/B48/B51-B53 子页面, D12-D15/D19-D20 Modal）

---

## 维度 1: 页面生命周期（Init / Destroy）

对每个页面回答：

### 1a. 页面加载（Init）
- 进入此页面时，调用哪些 API？调用顺序是什么（并行 or 串行）？
- 每个 API 的加载态如何展示？（skeleton / spinner / 占位文字）
- API 返回前页面处于什么状态？用户能看到什么、能操作什么？
- API 失败时页面如何降级？（error state / retry 按钮 / fallback 数据）
- 是否有路由守卫？（例: 已有 Community 时不应看到 B09 Empty 页）
- 是否从 URL query/params 读取初始状态？（例: `?tab=active`、`:id`）
- 是否从前一页面携带数据？（例: Wizard Step 1 的选择传入 Step 2）

### 1b. 页面离开（Destroy）
- 离开此页面时是否需要确认？（例: 表单有未保存修改 → "Unsaved changes" 弹窗）
- 是否有定时器/WebSocket 需要清理？（例: B15 的 integration-ping WebSocket）
- 草稿是否自动保存？保存到哪里？（localStorage / API draft）
- 离开后再返回，页面状态是否保持？（例: 筛选条件、展开状态、滚动位置）

---

## 维度 2: 按钮/操作 可见性与可用性

对页面上每一个按钮、链接、操作入口回答：

### 2a. 展示条件（Visibility）
- 该按钮在什么条件下显示？什么条件下隐藏？
- 是「条件隐藏」还是「始终显示但 disabled」？
- 角色/权限是否影响可见性？（Admin / Editor / Member）
- 订阅状态是否影响可见性？（Free vs Pro 功能区分）

### 2b. 可点击条件（Enabled / Disabled）
- 按钮 enabled 的前提条件是什么？（例: 表单校验通过、至少选 1 项、API 未在加载中）
- disabled 时的视觉表现？（opacity 0.5 + cursor not-allowed？灰色？）
- disabled 时是否有 tooltip 说明原因？（例: "Select at least one widget"）
- 是否有 loading 状态？（例: 点击后变 spinner，防止重复点击）

### 2c. 点击后效果（Action）
- 点击触发什么操作？（API 调用 / 路由跳转 / 弹窗 / 内联展开 / 下载 / 剪贴板）
- 如果是 API 调用：请求方法 + 端点 + 关键参数 + 成功响应处理 + 失败处理
- 如果是路由跳转：目标页面 + 携带参数 + 是否新标签
- 如果是弹窗：哪个 Modal（D 编号）+ 预填数据
- 操作成功后的反馈？（toast / 列表刷新 / 页面跳转 / 状态变更）
- 操作失败后的反馈？（toast error / 表单保留 / retry 选项）
- 是否有确认步骤？（confirm dialog → 再执行）
- 是否触发 D20 Publish Readiness Check？

---

## 维度 3: 页面导航（Forward / Backward）

对每个页面回答：

### 3a. 前进路径
- 从此页面可以前进到哪些页面？通过什么操作触发？
- 前进时携带什么数据/状态？
- 是否有条件性前进？（例: Wizard Next 按钮根据选择跳不同 Step 2 变体）

### 3b. 后退路径
- 此页面的「返回」按钮指向哪里？
- Breadcrumb 导航路径是什么？每段可点击？
- 浏览器 Back 键的行为是否与返回按钮一致？
- Wizard 中 Back 是否保留已填数据？

### 3c. 侧栏/全局导航
- 侧栏当前高亮项是什么？
- 侧栏子菜单展开状态是否正确？（Community 5 section headers / WL 展开子菜单）
- 面包屑层级是否与侧栏一致？

---

## 维度 4: 数据变化与刷新

对每个页面上展示的数据回答：

### 4a. 数据源
- 每个数据区块的 API 来源是什么？
- 数据是页面加载时一次性获取，还是分区块懒加载？
- 是否有轮询/WebSocket 实时更新的数据？更新频率？

### 4b. 数据变化触发
- 用户在此页面执行操作后，哪些数据区块需要刷新？
- 刷新方式：重新调 API（refetch）还是前端乐观更新（optimistic update）？
- 乐观更新失败时如何回滚？

### 4c. 跨页面数据一致性
- 在此页面修改数据后，其他已打开的页面是否需要感知？如何感知？
- 返回列表页时，列表是否自动刷新以反映刚才的修改？
- 统计数字（Stats Row）是否在子操作后实时更新？

### 4d. 缓存与过期
- API 响应是否有缓存？缓存时长（参考 §16 API 汇总的 Cache 列）？
- 用户手动刷新页面后是否获取最新数据？
- 是否有 stale-while-revalidate 策略？

---

## 维度 5: B端操作 → C端影响（CASCADE）

对每个 B端操作回答：

### 5a. 即时影响
- 此 B端操作完成后，C端（用户侧社区页面）会立即发生什么变化？
- 变化的传播机制是什么？（API 实时 / CDN 缓存 TTL / WebSocket 推送）
- C端用户正在浏览时，是否需要刷新才能看到变化？

### 5b. 影响范围
- 这个操作影响 C端的哪些页面/组件？列举。
- 是否影响 C端的导航/Tab 可见性？（例: 禁用模块 → C端对应 Tab 消失）
- 是否影响 C端用户的积分/等级/权益？

### 5c. 特殊级联
- 模块启用/禁用 → C端对应功能的出现/消失
- 任务发布/下架 → C端任务列表变化
- 积分规则变更 → C端排行榜/等级实时变化
- 商品上架/售罄 → C端商店显示变化
- 品牌设置变更 → WL 下 C端视觉主题变化（传播延迟？）
- Rule Builder 规则启停 → 链上事件奖励的开始/停止
- Privilege 层级变更 → C端用户权益的获得/失去

## 维度 6: 状态完备性

对每个页面检查以下状态是否都有明确规格：

| 状态 | 检查点 |
|------|--------|
| **正常态** | 有数据、功能正常时的完整 UI 规格 |
| **空数据态** | 无记录时的展示（empty illustration + 引导文案 + CTA） |
| **加载态** | API 请求中的 skeleton/spinner 规格 |
| **错误态** | API 失败 / 网络断开 / 500 错误的展示 + 恢复操作 |
| **权限不足态** | 角色无权限时的展示（按钮 disabled / 页面 403 / 提示升级） |
| **订阅限制态** | 超出配额时的展示（功能锁定 + 升级引导） |
| **并发态** | 多人同时操作 / 多标签页 / 重复提交 的处理 |
| **极端数据态** | 超长文本 / 超多条目 / 零值 / 负值 的展示 |

## 审计优先级定义

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| **P0** | 阻塞开发 | 开发者无法确定实现方式，必须产品确认后才能动手 |
| **P1** | 影响体验 | 可以先按常规实现，但可能导致返工 |
| **P2** | 完善度 | 不影响核心功能，属于锦上添花 |
